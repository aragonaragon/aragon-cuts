## Architecture in three sentences

The **frontend** is a single-window React 19 app in `src/`. The **backend**
is Rust in `src-tauri/src/` that exposes Tauri commands (`check_nvenc`,
`probe_video`, `encode_short`, `generate_thumbnails`) and emits typed
events (`encode:progress`). Heavy lifting (probe, thumbnail extraction,
encode) is done by **bundled FFmpeg/ffprobe sidecars** invoked through
`tauri-plugin-shell`.

State lives in React (Zustand was added as a dep but never used — every
piece of state is a `useState` in `App.tsx`). Settings persist via
`localStorage` with the prefix `shorts-maker:`.

## The encode pipeline — what every part does

```
[input file]
  ↓
ffmpeg -y -hide_banner -nostats -progress pipe:2 -stats_period 0.5 -fflags +genpts
  [-hwaccel cuda]                   ← only if use_nvenc == true (CRITICAL for AV1 4K)
  -ss <inSec>
  -i <path>
  -t <durationSec>
  -filter_complex
    [0:v]format=yuv420p,setpts=PTS-STARTPTS,split=2[orig][bg];
    [bg]scale=1080:1920:force_original_aspect_ratio=increase,crop=1080:1920,gblur=sigma=20[bgblur];
    [orig]scale=1080:-2[fg];
    [bgblur][fg]overlay=(W-w)/2:(H-h)/2,format=yuv420p
    [, drawtext for channel name watermark — top center of blurred area]
    [, drawtext for hook text — bottom center of blurred area, with alpha fade]
    [vout]
  -map [vout] -map 0:a?
  -pix_fmt yuv420p
  -c:v h264_nvenc -preset p5 -cq 23   (or libx264 -preset medium -crf 20)
  -c:a copy                            (or -c:a aac -b:a 192k -ac 2 fallback)
  -movflags +faststart
  <output>
```

Output goes to `<source_dir>/<source_stem>_shorts/clip_<inSec>-<outSec>s.mp4`,
with auto-numbering on collisions.

### Why each piece is there (don't rip them out without thinking)

- **`-hwaccel cuda`** when NVENC is in use — bypasses the bundled
  `libaom-av1` decoder which refuses AV1 streams at Level 6
  (`seq_level_idx=23`), the level NVIDIA's own AV1 encoder writes for 4K
  60fps. This was the v0.2.1 fix. Without it, NVIDIA-recorded gameplay
  fails with `-22 Invalid argument` from h264_nvenc (which is misleading —
  the real failure is in the AV1 decode stage, not the encode stage).
- **`format=yuv420p` at the start and end of the filter chain** — strips
  10-bit / HDR sources down to 8-bit 4:2:0 so NVENC doesn't reject the
  frames. NVENC's H.264 path is 8-bit only.
- **`setpts=PTS-STARTPTS`** — resets timestamps after `-ss`. NVENC dislikes
  huge or negative PTS values it sometimes gets when the source is
  AV1 + container offsets are weird.
- **`-pix_fmt yuv420p` placed before `-c:v`** — makes NVENC negotiate its
  input format correctly from initialization. After the encoder is wrong.
- **NVENC args are deliberately minimal** (just `-preset p5 -cq 23`). The
  full `-rc vbr -maxrate 12M -bufsize 24M -tune hq` recipe from the
  original plan caused encoder init failures on some sources. Don't add
  these back without testing on AV1 4K HDR.
- **Audio codec fallback** (`audio_codec_args` in `ffmpeg.rs`): the MP4
  muxer can't carry PCM, FLAC, Vorbis, etc. If the source's audio codec
  is in the "MP4-compatible" allowlist (AAC, MP3, AC3, EAC3, ALAC, Opus,
  mp4a) we `-c:a copy`. Otherwise we re-encode to AAC 192k. The user is
  fine with losing PCM losslessness for the sake of not failing.

## Progress reporting (this is non-obvious)

FFmpeg's default progress line on stderr uses `\r` as a separator, not
`\n`. Tauri's shell plugin reads stderr line-by-line on `\n`, so by default
you get one giant chunk at the end and no progress updates.

The fix (already in place): pass `-progress pipe:2 -stats_period 0.5
-nostats` to FFmpeg. That writes structured key=value lines every 500 ms
to stderr, separated by real newlines. Rust then parses
`out_time_us=<n>` and `speed=<n>x`, accumulates them, and emits a typed
`encode:progress` Tauri event when it sees `progress=continue` or
`progress=end`. The frontend listens via
`listen<EncodeProgressEvent>("encode:progress", ...)` and patches the
matching clip's status with `{ kind: "encoding", percent, speed }`.

If you ever rip out the `-progress pipe:2` flag, the progress bars stop
updating. Don't do that without replacing the mechanism.

### Shipping a release (the user's exact pattern)

```powershell
# 1) Bump version in THREE files
#    - package.json:           "version": "0.X.Y"
#    - src-tauri/Cargo.toml:   version = "0.X.Y"
#    - src-tauri/tauri.conf.json: "version": "0.X.Y"

# 2) Commit (use a file for the message — PowerShell heredocs corrupt git
#    commit messages that contain certain chars)
"...message..." > $env:TEMP\msg.txt
git -c user.name="aragon" -c user.email="nathoool92@gmail.com" commit -F "$env:TEMP\msg.txt"

# 3) Push
git push origin main

# 4) Build (~3-5 min on first run after deps change)
Stop-Process -Name "shorts-maker","cargo","rustc" -Force -ErrorAction SilentlyContinue
pnpm tauri build

# 5) Delete the previous release + tag (user prefers ONE visible release)
gh release delete v<previous> --yes --cleanup-tag

# 6) Write release notes to a file, then create
Set-Content $env:TEMP\notes-vX.Y.Z.md "..."
gh release create v0.X.Y `
  "src-tauri\target\release\bundle\nsis\Shorts Maker_0.X.Y_x64-setup.exe" `
  "src-tauri\target\release\bundle\msi\Shorts Maker_0.X.Y_x64_en-US.msi" `
  --title "Shorts Maker v0.X.Y" `
  --notes-file "$env:TEMP\notes-vX.Y.Z.md" `
  --latest
```

**Why use a notes file** — PowerShell's `@'...'@` here-string can be passed
to `gh release create --notes "..."` but characters like `·`, `/`, `→`,
and apostrophes inside contractions trip up the shell parser. A notes
file dodges all of that.

## Gotchas — things that took us hours to figure out

1. **AV1 Level 6 from NVIDIA Record** breaks libaom-av1 decode. Fix:
   `-hwaccel cuda` before `-i` when NVENC is in use. Error message is
   misleading — looks like a NVENC encoder failure (`-22 Invalid
   argument`) but is actually a decoder failure earlier in the pipeline.
   This is **the** lesson of v0.2.1.

2. **`pnpm tauri dev`'s wrapper sometimes exits with 4294967295** seconds
   after launching the binary. The binary itself is fine — it's the
   wrapper that's flaky. Use the two-process pattern (Vite + direct exe)
   for iteration.

3. **Webview2 + offline install** — `webviewInstallMode: offlineInstaller`
   in `tauri.conf.json` adds ~120 MB to the installer but is the only way
   to guarantee install on a machine without internet. Don't switch to
   `downloadBootstrapper` unless the user explicitly says size matters
   more than offline-readiness.

4. **`glib` 0.18 in `Cargo.lock`** — Socket.dev / Dependabot will warn
   about a CVE in this crate. It's a Linux-only transitive dep (via
   `tao` → `muda` → `gtk`) and is NOT compiled into our Windows builds
   (`cargo tree -i glib --target x86_64-pc-windows-msvc` returns "nothing
   to print"). Document as "not affected" if asked.

5. **`drawtext` font path on Windows** — must be
   `'C\:/Windows/Fonts/arialbd.ttf'` with the colon escaped via `\:`.
   See `FONT_PATH` constant.

6. **`text_shaping=1`** is required on every `drawtext` filter for Arabic
   to render with proper letter joining. Currently set on all three
   drawtext variants (watermark + hook). Leave it on.

7. **Tauri 2 sidecar invocation** — use `app.shell().sidecar("ffmpeg")`,
   NOT `binaries/ffmpeg`. The binary file must be named with the
   `-<TARGET_TRIPLE>` suffix (`ffmpeg-x86_64-pc-windows-msvc.exe`) and
   referenced in `tauri.conf.json` `bundle.externalBin` as
   `"binaries/ffmpeg"` (without the suffix).

8. **`tauri-plugin-shell` line-based stderr** — see *Progress reporting*
   above. FFmpeg's `\r`-separated default progress doesn't work.

9. **CRLF warnings on every git commit** — Windows + git default
   autocrlf. Harmless. Ignore.

## What's deliberately NOT in the project (user rejected these)

The user shut down each of these explicitly:

- **Whisper / auto-captions** — too heavy (~500 MB model), gameplay shorts
  don't need it, captures-only audio doesn't benefit. Was in the original
  plan, was dropped.
- **Additional output styles** (Center Crop, Fit with Bars, Pan & Scan) —
  the sidebar UI for these existed briefly, was deleted. The user
  prefers the single Blurred BG look.
- **Settings panel** — no output-folder picker, no force-CPU toggle, no
  default style selector. Channel name + hook text + watermark style are
  the only configurables.
- **Auto-updater / telemetry / cloud anything** — explicitly against the
  "offline-first, no surprises" stance.
- **macOS / Linux builds** — Windows-only for v0.x. The codebase is
  cross-platform-friendly in principle (Tauri 2 supports all three) but
  no sidecar binaries or font paths are set up for the other OSes.

If the user asks for any of these later, treat as a new conversation and
push back gently: "we scoped this out at v0.2.x because..."

## A note on security alerts

If GitHub / Socket / Dependabot flags something:

- Check whether the package is actually in the **Windows-target** dep
  tree with `cargo tree -i <package> --target x86_64-pc-windows-msvc`.
  Lots of Tauri's transitive deps are Linux-only and Cargo.lock lists
  them anyway. Scanners don't know the difference.
- The user does not want to chase non-applicable CVEs.
- If a CVE genuinely applies, bump it and ship a patch release.
